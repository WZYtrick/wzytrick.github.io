Relation Databases

基本概念

关系型数据库是一个表的概念，表头成为属性(attribute),一行为一个元组(tuple)，一个关系模式(relation schema)表示为R=(A1,A2,A3)(或表示为r(R))其中R是关系名，A123是属性，每个属性有它的域(domain)，就是它的范围。域是这个属性在这个关系中所有取值的集合，若A1域为D1,A2为D2，A3为D3则有该relation为D1,D2,D3的一个子集。

​ tips:

  • relation是原子的(atomic)也就是例如两条记录
name acount
jack A101
jack A102

​ 其中两项不合并。

  • 记录是无序的

  • 一般称某一个元组时，用元组变量(Tuple variable)来表述例如，

    t[name]=jack这条记录(元组)

keys

keys是r(R)的真子集

  1. super key 超码
    • 若任意两tuple在这一个或一组attributes上可被区分，那么定义这一个或一组attribute为超码。
  2. candidate key 候选码
    • 作为超码的子集，候选码是最小的超码，它的子集必不是超码，例如姓名，ID是一组超码，而不是候选码，因为其子集ID可以作为超码。
  3. primary key 主码
    • 主码是候选码中的一个属性，它作为一个最特殊的属性，主要用于被查询的索引，例如身份证号，它具有不易变性。
  4. foreign key 外码
    • 设有关系r1,其中的一个属性A1是另一关系r2的primary key，则称r1的这个属性为外码。其中，r1称为引用关系(referencing relation)，r2称为被引用关系(referenced relation)
    • 值得注意的是，这里有一个Referential Integrity Constraints,它的意思是，如果这个属性是外码，那么他的值要么为空，要么在其引用的关系中有完整的定义。例如，银行中关系客户(姓名，电话号，账户，….)其中账户为外码，对应关系账户(姓名，账户，存款，…)中的主码账户(因为一个人可以有多个银行账户)，而若客户关系中存在账户而账户关系中并没有存储就造成了错误。所以一定要保证先在账户中增添信息后再于客户中添加账户信息。
