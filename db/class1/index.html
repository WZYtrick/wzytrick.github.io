DataBase System

​ 数据库系统(DBS)包括数据库(DB)以及访问数据的程序(Database Management System,DBMS,数据库管理系统)，DBMS是系统软件。

Drawbacks of using file system without DBMS

  1. Data redundancy and inconsistency(数据的冗余与不一致)

    信息充分存储，信息量没有增加。冗余数据的出现，导致修改了一个数据而其他相同冗余数据没有被修改。

    e.g. 在银行app中，不同语言写的不同程序以不同的格式保存着同样的信息，顾客的姓名电话号信息被保存于“保存账户” “检查账户”两个程序中。

  2. Difficulty in accessing data(数据访问困难)

    数据信息存储不规范，每次任务都需重新编写程序

    e.g. 需要年龄五十岁以上顾客信息。需要xxx信息。

  3. Data isolation(数据孤立)

    数据存储在多个可能格式不同的文件中

    e.g. 由于程序数据存储的格式不同，在访问时，例如想要找到在某一地区的账户，需要访问不同格式的数据，十分麻烦，需要编写程序分别访问不同格式文件。

  4. Integrety problems(完整性问题)

    对于数据存在的约束，如银行账户余额不能为负，这类约束一般在程序中体现，而当新加入新约束时很难通过修改程序实现。当要添加有关多个文件中数据的约束时会更加困难。

  5. atomicity problem(原子性问题)

    e.g.银行存款时一方付款后由于系统故障另一方没有接受到付款

  6. concurrent-access anomaly(并发访问异常)

    e.g.两个用户同时取一个账户的存款，结果有问题。

  7. security problem(安全性问题)

    并非所有数据库系统用户都可以访问所有数据。

Abstract view of data

​ 与计算机其他课程的思想相同，数据库系统也采用抽象的分层结构来隐藏底层的细节。

  • Physical level
  • Logical level
  • View level

Physical level:

​ 一条记录如何存储

Logical level:

​ 什么数据存储在数据库以及数据之间的关系；整个数据库。

View level:

​ 给用户看的数据库的一部分，屏蔽掉用户不需要知道或不应该知道的信息。

Instances and Schemas

  • Schemas 模式

    the overall design of the database 总体设计(就像程序中的变量)

  • Instance 实例

    the collection of information stored in the data base at aparticular point in time. 在某一时刻数据库中信息的集合。(就像程序中的变量的值)

Schemas of DBS

  • Physical schemas
  • Logical schemas
  • Sub schemas

每层相互独立，修改某一层的模式不需要修改其他层。

Data Models

  • Relational model 关系模型

    使用表的集合来表示数据之间的联系，每一行为一个记录，每列是一个属性/字段(有唯一列名)，它是基于记录的的模型。

  • Entity-Relationship data model 实体-联系模型

    ER模型是对现实世界的一种抽象 例如下图

    其中菱形表示关系，长方形表示抽象出来的实体，椭圆表示实体所拥有的属性。

  • Object-based data models 基于对象的数据模型

    它基于ER模型，增加了封装，方法(函数)，对象标识符等概念。

  • Semistructured data model 半结构化数据模型(e.g. XML)

  • Other older models:

    • Network data model 网状数据模型
    • Hierarchical data model 层次数据模型
